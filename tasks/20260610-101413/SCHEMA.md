# OpenCode HTTP API — event schema spike

Companion to `TASK.md`. Captures the actual wire shape of `opencode
serve`'s HTTP/SSE API (v1.15.13) so the new `AgentManager` can be
written against real behaviour, not the SDK's documented surface.

Generated by spinning up `opencode serve --port 18765` against this repo
and capturing live events for two prompts (a plain question and a
bash-tool prompt). Raw captures are in `/tmp/nix-shell.*/opencode/spike/`
during the spike but are not committed.

---

## TL;DR for the implementer

1. **Chat is non-streaming.** `POST /session/{id}/message` blocks until
   the model finishes, then returns the full assistant reply.
2. **Streaming events come from a *separate*, *server-global* SSE bus**:
   `GET /event`. One stream per `opencode serve` instance; events for
   every session are interleaved on it. Filter by `properties.sessionID`.
3. **The Python SDK (`opencode_ai==…`) is partly stale.** Several event
   types and several `Session` fields are emitted by the live server but
   missing from the SDK's Pydantic models. **Do not** use the SDK to
   parse the event stream — go raw httpx + JSON dicts. The SDK is fine
   for one-shot REST calls (create/delete/list session) since
   `_strict_response_validation=False` is the default.
4. **`message.part.delta` is the per-token text streaming event** —
   not `message.part.updated` (which only sends cumulative snapshots).
5. **`session.idle` is the assistant-turn-finished signal.** Use it to
   close the stream and resolve the request.
6. **Port: not standardised.** TASK.md says `8080`, the existing
   `utils/tools/opencode_tool.py` says `4096`, the SDK fallback is
   `localhost:54321`, and `opencode serve --port 0` (the binary's
   default) means "OS-assigned random". Action: set `--port` explicitly
   in the systemd/Nix unit and read it via `OPENCODE_BASE_URL`. The TASK
   port figure is wrong — fix when wiring the daemon.

## REST endpoints used by the new runtime

| Method | Path | Purpose | SDK call |
|---|---|---|---|
| POST | `/session` | create session (empty body OK) | `client.session.create()` |
| GET  | `/session` | list sessions (filtered by `projectID` server-side) | `client.session.list()` |
| GET  | `/session/{id}` | session detail | not in SDK |
| DELETE | `/session/{id}` | delete session | `client.session.delete(id)` |
| POST | `/session/{id}/abort` | cancel an in-flight turn | `client.session.abort(id)` |
| POST | `/session/{id}/message` | **send chat (non-streaming, blocking)** | `client.session.chat(id, ...)` |
| GET  | `/session/{id}/message` | list messages in a session | `client.session.messages(id)` |
| GET  | `/event` | **SSE event stream (server-global)** | `client.event.list()` |
| GET  | `/config/providers` | provider + model catalog | (different from `client.app.providers`) |
| GET  | `/app` | app metadata — *non-JSON or 404 in 1.15.13* (skip) | `client.app.get()` |

`POST /session/{id}/message` body:

```json
{
  "providerID": "github-copilot",
  "modelID": "claude-sonnet-4",
  "parts": [{"type": "text", "text": "<user input>"}],
  "system": "<optional system prompt for this turn>",
  "mode": "<optional mode name>",
  "tools": {"bash": true, "edit": false}
}
```

Response (real, not SDK-typed): `{ "info": <AssistantMessage>, "parts": [<Part>, …] }`.
The SDK's `chat()` returns just `AssistantMessage`; in 1.15.13 the
server actually returns the wrapped form. Treat the SDK's typed return
as a hint, not truth.

## Session schema — what the live server emits

The SDK's `Session` model knows: `id`, `time`, `title`, `version`,
`parentID?`, `revert?`, `share?`. The live server *also* sends:

- `slug` — human-friendly session id ("sunny-sailor")
- `projectID` — hash of the project root directory
- `directory` — absolute path of the project root
- `path` — sub-path within project
- `cost` (float), `tokens` (`{input, output, reasoning, cache: {read, write}}`)
- `summary` (`{additions, deletions, files}`)
- `agent` (e.g. `"build"`)
- `model` (`{id, providerID, variant?}`)
- `permission` — list of `{permission, pattern, action}` rules

The proxy endpoints (`/v1/opencode/sessions`, `/v1/opencode/sessions/{id}`)
should pass JSON through verbatim; do not deserialise via the SDK type.

## Event stream — observed types (v1.15.13)

Each SSE message has shape:

```
data: {"id":"evt_<…>","type":"<EVENT_TYPE>","properties":{…}}
```

*No* `event:` line. Events have an outer `id` the SDK type doesn't model.

### Events we observed during a normal turn (50 events for one tool prompt)

| Type | Modeled by SDK? | Observed for | Notes |
|---|---|---|---|
| `server.connected` | yes | first event after subscribing | `properties: {}`. Heartbeat / handshake. |
| `session.next.agent.switched` | **no** | once per turn before `message.updated` | Tells us which agent (`"build"` etc.) handled this turn. |
| `session.next.model.switched` | **no** | once per turn | `{model: {id, providerID, variant}}`. |
| `message.updated` | yes | both user + assistant message lifecycle | `properties.info: Message` (full snapshot). Fires several times per assistant message as state changes (system prompt added, agent set, tokens accrue, `time.completed` set). |
| `message.part.updated` | yes | every part state change | Full part snapshot (text, tool, step-start, step-finish, file, snapshot, patch). For `tool` parts the embedded `state.status` walks `pending → running → completed | error`. **Not** delta-style for text. |
| `message.part.delta` | **no** | per text chunk during assistant streaming | `{sessionID, messageID, partID, field: "text", delta: "<chunk>"}`. **This is the event the chat-stream renderer should consume for `kind="text"`.** |
| `message.part.removed` | yes | not observed in spike | Deletion of a part (revert flow). |
| `session.status` | **no** | several per turn | `{status: {type: "busy" | "idle" | …}}`. Coarse-grained. |
| `session.diff` | **no** | end of turn (file-edit turns) | `{diff: [<patch>, …]}`. Empty `[]` for non-edit turns. |
| `session.updated` | yes | tokens/cost accrual | `properties.info: Session`. |
| `session.idle` | yes | **end-of-turn** | `{sessionID}`. **Use this as the stop signal.** |
| `session.error` | yes | not observed in spike | `{sessionID?, error: ProviderAuthError | UnknownError | MessageOutputLengthError | MessageAbortedError}`. Discriminator is `error.name`. |
| `session.created` | **no** | on `POST /session` | Carries new `info: Session`. |
| `session.deleted` | yes | not observed in spike | On `DELETE /session/{id}`. |
| `permission.updated` | yes | not observed in spike | `{id, sessionID, title, metadata, time}`. Fires when a tool call needs user approval. Behaviour depends on session-level `permission` rules. |
| `installation.updated` | yes | not observed | OpenCode binary self-update. Ignore. |
| `lsp.client.diagnostics` | yes | not observed | LSP integration. Ignore for our use. |
| `storage.write` | yes | not observed | Internal storage. Ignore. |
| `file.edited` | yes | not observed in this spike, but emitted when assistant edits a file | `{file: "<path>"}`. |
| `file.watcher.updated` | yes | not observed | `{file, event: "rename" | "change"}`. Ignore. |
| `ide.installed` | yes | not observed | Ignore. |

### Tool part state machine (verified live)

A single tool call produces ~5 `message.part.updated` events:

```
state.status = "pending"   →   tool not yet running, only callID known
state.status = "running"   →   state.input populated, state.title sometimes still empty
state.status = "running"   →   state.input + state.title populated
state.status = "running"   →   (more updates as title is refined)
state.status = "completed" →   state.output, state.metadata, state.time.end set
```

For `bash` tool, `state` looks like:

```json
{
  "status": "completed",
  "input": {"command": "echo hello", "description": "Prints hello to stdout"},
  "output": "hello\n",
  "metadata": {"output": "hello\n", "exit": 0, "description": "...", "truncated": false},
  "title": "Prints hello to stdout",
  "time": {"start": …, "end": …}
}
```

The `metadata` may contain a giant provider-specific blob (e.g.
`copilot.reasoningOpaque` is several KB of base64). **Strip that before
logging or persisting.**

### Sample: text streaming via `message.part.delta`

```json
{
  "id": "evt_…",
  "type": "message.part.delta",
  "properties": {
    "sessionID": "ses_…",
    "messageID": "msg_…",
    "partID": "prt_…",
    "field": "text",
    "delta": "Hi there"
  }
}
```

In our 11-delta tool turn, all deltas had `field: "text"`. The SDK
schema would accept other field values for non-text streaming (e.g.
reasoning) but we did not observe any in this spike.

## Mapping → `ThinkingEvent`

Target: `utils/callbacks.py::ThinkingEvent`. The `ToolCallbackHandler`
class is deleted; replaced by a stateless function:

```python
def map_opencode_event(
    raw: dict,
    *,
    target_session_id: str,
    target_message_id: str | None,  # filled in once we observe the assistant message
) -> ThinkingEvent | None: ...
```

Caller is responsible for: filtering by `sessionID`, tracking the
`messageID` of the in-flight assistant turn, and detecting the
terminator (`session.idle` for our session, or `session.error`).

| Source event | Condition | → `ThinkingEvent` | Notes |
|---|---|---|---|
| `message.part.delta` | `properties.field == "text"` | `kind="text", source="scufris", text=delta, depth=0` | The streaming text. Append-style; CLI already concatenates. |
| `message.part.updated` | `part.type == "tool"` and `part.state.status == "running"` and we haven't emitted for this `part.id` yet | `kind="tool_call", source="scufris", text=part.tool, arg=part.state.title or part.state.input.<best>, depth=0` | First-time-running. Dedup by `part.id`. `arg` falls back to a one-line summary of `state.input` when `title` is empty. |
| `message.part.updated` | `part.type == "tool"` and `part.state.status == "completed"` | `kind="tool_result", source="scufris", text=part.tool, depth=0` | Currently suppressed by default in renderer; preserve that policy. |
| `message.part.updated` | `part.type == "tool"` and `part.state.status == "error"` | `kind="tool_result", source="scufris", text=f"{part.tool} failed: {state.error}", depth=0` | Surface tool errors in trace. |
| `message.part.updated` | `part.type == "text"` | nothing | Redundant with `message.part.delta`. Skip. |
| `message.part.updated` | `part.type` in (`step-start`, `step-finish`, `snapshot`, `patch`, `file`) | nothing | No user-facing value. (Maybe `tool_meta` with cost/tokens for `step-finish` later.) |
| `permission.updated` | sessionID matches | `kind="tool_meta", source="scufris", text=f"permission: {title}"` | Only meaningful if interactive. v1: configure session-level permission rules at create-time so this never fires (auto-allow). |
| `session.idle` | sessionID matches | **terminator** (return `None`, signal end-of-stream) | Final assistant text is reconstructed from accumulated `delta`s, OR by reading `GET /session/{id}/message` once. |
| `session.error` | sessionID matches | **terminator + raise** | Map `error.name` to a user-friendly string. Caller raises an exception that becomes a 5xx in `/v1/chat`. |
| `session.deleted` | sessionID matches | terminator + signal "stale" | `AgentManager` retries with a freshly-created session (per TASK acceptance criterion). |
| `message.updated` | any | nothing | We don't surface message metadata; useful only as a sanity check. |
| `session.status`, `session.diff`, `session.updated`, `session.created`, `session.next.*`, `server.connected`, `installation.*`, `lsp.*`, `storage.*`, `file.watcher.*`, `ide.*` | — | nothing | Ignore. |
| `file.edited` | sessionID matches | optional `kind="tool_meta", text=f"edited {file}"` | Nice-to-have v2. |

### `source` semantics

In the LangChain era, `source` was a sub-agent name (`coding_agent`,
`knowledge_agent`, …) so the renderer could indent traces by depth.
With OpenCode + skills, the agent hierarchy is gone — there's a single
assistant. Recommendation:

- `source = "scufris"` for assistant text and tool_calls — flat depth=0.
- Drop the `is_sub_agent` / depth-indent logic. Renderer keeps the
  field but always sees depth=0.
- Re-introduce nesting later if/when OpenCode "task tool" sub-sessions
  show up in traces (events would have a different `sessionID`,
  filterable by parentID).

### Final-reply extraction

`AgentManager.process_message` must return a single string. Two options:

- **(a) Accumulate deltas during streaming.** Concatenate every
  `message.part.delta.delta` whose `field == "text"` and (eventually)
  whose `messageID == <our assistant msg>`. Final string is ready when
  `session.idle` arrives.
- **(b) After `session.idle`, fetch `GET /session/{id}/message` and pick
  the latest assistant message's text parts.**

Recommendation: **(a)** — it's free (we're already iterating events),
deterministic, and matches the user-visible stream byte-for-byte.

## Open questions / decisions before implementation

1. **One global `/event` connection or one per request?**
   - Global: single long-lived task fans events into per-`sessionID`
     asyncio queues. Saves N connections under load; needs a fan-out
     layer with cleanup.
   - Per request: open `/event`, fire `POST /session/{id}/message`,
     read until `session.idle`, close. Simpler; one extra HTTP
     connection per chat. Fine for v1 (low concurrency, single host).
   - **Recommendation:** per request for v1. File a follow-up task to
     consolidate to a global listener if/when concurrency matters.

2. **Permission rules.** The session JSON includes a `permission` array
   that can deny/auto-allow specific tools. Existing stored sessions
   (e.g. `…@explore subagent`) have `task: deny`, `todoread: deny`,
   `todowrite: deny`. Scufris running as a personal-assistant turn-loop
   needs everything *allowed* (no human in the loop) — but also needs
   protection against destructive commands (skills will gate that). v1
   policy: configure session-level `permission: []` (no rules → default
   allow) and lean on the eventually-shipped skills to constrain
   behaviour. Document the tradeoff.

3. **`mode` parameter on chat.** Defaults to `"build"` (observed). Other
   modes (`"plan"` etc.) load different prompts/tools. v1: leave as
   default; revisit when skills land.

4. **Tools allow-list.** `POST /session/{id}/message` accepts
   `tools: Dict[str, bool]`. Useful for stripping the model's access to
   `task` (sub-agent spawning) and similar power tools that we don't
   want firing through the personal-assistant surface. v1: send
   `{"task": false, "todowrite": false, "todoread": false}` to match
   the existing sub-session convention. Re-evaluate.

5. **Session reuse vs. recreate.**
   - Pinned by TASK: one persistent session per `user_id` until cleared.
   - Spike confirms: sessions persist across server restarts (we saw
     20+ historic sessions for this directory). So `_sessions: Dict[int,
     str]` rebuilt fresh on restart will leave orphan sessions on the
     OpenCode side. Probably fine — they're reachable via list and can
     be manually pruned. File a follow-up task for "persistent session
     id ↔ user_id store" (already in TASK's "Out" section).

6. **Token / cost accounting.** Live `session.updated` and
   `step-finish` events carry per-turn cost and token counts. The
   existing `/v1/stats` endpoint sources its data from
   `ChatHistoryManager`; switching to OpenCode's numbers is a clean
   v2 task — out of scope for the runtime swap.

7. **Multi-surface sharing.** TASK already calls this out: one session
   per user means CLI + Telegram messages interleave in a single
   conversation. Spike doesn't change the answer — it's still desirable.

8. **Upstream SDK staleness.** Fine for now (we're going raw httpx for
   the event stream and synchronous calls only need create/delete/list).
   File a follow-up if the SDK type drift starts hurting.

## Implementation recommendations

1. **`utils/opencode_client.py`** (new):
   - Use `httpx.AsyncClient(base_url=…)` directly. **Do not** wrap the
     SDK — its types lie about the wire format and its sync clients
     don't compose with `asyncio` cleanly.
   - Methods:
     ```python
     async def create_session(self) -> dict: ...
     async def delete_session(self, sid: str) -> None: ...
     async def list_sessions(self) -> list[dict]: ...
     async def chat_stream(
         self,
         sid: str,
         prompt: str,
         *,
         system: str | None,
         provider_id: str,
         model_id: str,
         tools: dict[str, bool] | None = None,
     ) -> AsyncIterator[dict]:
         # opens GET /event in parallel, posts the message,
         # yields raw event dicts filtered by sessionID,
         # terminates on session.idle / session.error.
     ```
2. **`utils/opencode_events.py`** (new):
   - `map_opencode_event(raw, *, target_session_id, target_message_id)`
     pure function returning `ThinkingEvent | None`.
   - `extract_assistant_text(deltas: list[str]) -> str` — concat helper.
3. **`utils/agent.py`**: `AgentManager.process_message` becomes:
   ```python
   sid = self._sessions.get(user_id) or await self._client.create_session()
   try:
       text_chunks: list[str] = []
       async for raw in self._client.chat_stream(sid, message, system=…):
           tev = map_opencode_event(raw, …)
           if tev: dispatch_event(tev)
           if raw.get("type") == "message.part.delta" and raw["properties"].get("field") == "text":
               text_chunks.append(raw["properties"]["delta"])
   except SessionGone:
       sid = await self._client.create_session(); … retry once
   self._sessions[user_id] = sid
   return "".join(text_chunks)
   ```
4. **`scufris_server/routes/chat.py`**: no behavioural change needed —
   it already serializes whatever `dispatch_event(ThinkingEvent)`
   produces and ends on the final reply text.
5. **Bypass the SDK entirely.** It buys us nothing here and its
   pydantic validation will reject the unknown event types.
